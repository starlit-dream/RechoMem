# AGENTS.md — RechoMem

**生成时间：** 2026-03-26  
**技术栈：** Rust 2021 · Tokio · rmcp 1.2 · SQLite (sqlx 0.8) · LanceDB 0.27 · async-openai 0.28

## 项目概述

RechoMem 是一个为 LLM 长期记忆设计的三层语义漏斗引擎，以 MCP stdio 服务形式对外暴露。所有记忆数据分三层存储，检索时按需逐层下探，以最低 Token 成本覆盖最宽的语义范围。

## 三层架构速查

| 层级 | 文件 | 存储介质 | 职责 |
|------|------|----------|------|
| L1 | `storage/vector.rs` | LanceDB（嵌入式） | Topic Embedding 向量索引，触发器 |
| L2 | `storage/sqlite.rs` | SQLite（sqlx async） | 摘要、实体反向索引，常态注入内容 |
| L3 | `storage/jsonl.rs` | JSONL 追加文件 | 原始对话全文，按 offset+length 下探 |

## 核心类型

| 类型 | 文件 | 用途 |
|------|------|------|
| `MemoryBlock` | types.rs | 完整 block 元数据，含 raw_offset / raw_length 指针 |
| `SummaryRecord` | types.rs | L2 轻量摘要，MCP search 工具的常态返回值 |
| `SearchResult` | types.rs | 含 semantic_score / entity_score / recency_score / final_score |
| `RawLogChunk` | types.rs | L3 原文片段，drill_down 工具的返回值 |
| `AppConfig` | config.rs | 所有路径与 OpenAI 参数，仅从环境变量读取 |
| `AppState` | mcp.rs | Arc 包裹的三个存储后端句柄，注入 MCP 服务 |

## MCP 工具接口

## 项目特有约束

**stdio 纯净原则**：任何日志、调试输出必须使用 `tracing`（写 stderr），绝不能写 stdout。MCP 客户端通过 stdout 通信，任何非协议输出会破坏帧结构。

**追加写优先**：L3 JSONL 文件只允许追加，禁止原地修改。block 的 raw_offset 是该 block 第一行的字节偏移，raw_length 是该 block 全部内容的字节长度。

**错误类型统一**：所有模块返回 `crate::error::Result<T>`，禁止在模块边界使用 `unwrap()` 或 `expect()`。MCP 工具函数内部将 `RechoMemError` 转换为 `rmcp::ErrorData`。

### 3.1 简单技术问题

**提问：** Go 的 `sync.Mutex` 和 `sync.RWMutex` 有什么区别？

**期望回复：**

`sync.Mutex` 是互斥锁，同一时间只允许一个 goroutine 持有锁，无论读写。

`sync.RWMutex` 在此基础上区分了读锁和写锁。多个 goroutine 可以同时持有读锁，但写锁是排他的——持有写锁时，其他 goroutine 既无法读也无法写。

适用场景：读多写少的共享数据结构适合使用 `RWMutex`，读写频率接近时 `Mutex` 反而更简单且开销更低。

### 3.2 方案讨论

**提问：** 消息推送用 WebSocket 还是 SSE？

**期望回复：**

两种方案的核心差异在于通信方向。

**WebSocket** 提供全双工通道，客户端和服务端可以随时互发消息。适合聊天、协同编辑等需要双向实时交互的场景。代价是连接管理更复杂，需要处理心跳、重连、状态同步。

**SSE（Server-Sent Events）** 是单向的服务端推送，基于 HTTP 长连接。实现简单，天然支持断线重连和事件 ID。适合通知推送、数据流等只需服务端向客户端发送消息的场景。

当前项目的消息推送属于服务端单向通知，SSE 在实现成本和可靠性方面更有优势。如果后续需要扩展为双向交互（例如已读回执），再迁移到 WebSocket 的改动量也可控。

### 3.3 被指正后的回复

**指正：** 这个写法有问题，`defer` 在循环里会导致资源泄漏。

**期望回复：**

确实，`defer` 的执行时机是函数返回时而非循环迭代结束时，放在循环内部会导致文件句柄持续累积直到函数退出。

修正方案是将循环体内的操作提取为独立函数，或者手动在每次迭代末尾调用 `Close()`：

```go
for _, path := range paths {
    f, err := os.Open(path)
    if err != nil {
        return err
    }
    // 处理文件内容
    f.Close()
}
